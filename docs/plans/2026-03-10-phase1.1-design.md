# Phase 1.1 Design: `tokenize()` + `deanonymize()`

Reversible redaction for LLM round-trip pipelines.

## Problem

`kloak.redact()` replaces PII with generic labels like `<EMAIL_ADDRESS>`. When multiple entities share a type, information is lost — you can't distinguish which email was which, and you can't restore the originals in an LLM response.

## Solution

Two new core API functions:

- `tokenize(text)` — replaces PII with numbered tokens (`<EMAIL_ADDRESS_1>`, `<EMAIL_ADDRESS_2>`) and returns a mapping from token to original value.
- `deanonymize(text, mapping)` — replaces tokens in text with original values using the mapping.

## Public API

```python
import kloak
from kloak import TokenizeResult

# Tokenize
result = kloak.tokenize("Email ahmad@mail.com, call 0123456789")
# TokenizeResult(
#   text="Email <EMAIL_ADDRESS_1>, call <PHONE_NUMBER_1>",
#   mapping={"<EMAIL_ADDRESS_1>": "ahmad@mail.com", "<PHONE_NUMBER_1>": "0123456789"},
#   entities=[EntityMatch(...), ...]
# )

# include/exclude work the same as redact()
result = kloak.tokenize(text, include=["EMAIL_ADDRESS"])
result = kloak.tokenize(text, exclude=["PERSON"])

# Persist mapping
result.to_json("mapping.json")      # write to file
json_str = result.to_json()          # or return string

# Deanonymize
mapping = TokenizeResult.load_mapping("mapping.json")
original = kloak.deanonymize(llm_response, mapping)

# Also works on KloakEngine
engine = KloakEngine(score_threshold=0.6)
result = engine.tokenize(text)
original = engine.deanonymize(llm_response, result.mapping)
```

## Types

```python
@dataclass(frozen=True)
class TokenizeResult:
    text: str                        # tokenized text
    mapping: dict[str, str]          # {"<EMAIL_ADDRESS_1>": "ahmad@mail.com"}
    entities: list[EntityMatch]      # positions reference original text

    def to_json(self, path: str | None = None) -> str | None:
        """Write mapping to JSON file, or return JSON string if no path."""

    @classmethod
    def load_mapping(cls, path: str) -> dict[str, str]:
        """Load mapping dict from JSON file."""
```

## Token Format

`<ENTITY_TYPE_N>` where N is a 1-indexed counter per entity type, assigned left-to-right by position in the text.

Example: `"email a@x.com and b@x.com"` → `"email <EMAIL_ADDRESS_1> and <EMAIL_ADDRESS_2>"`

## Implementation

### tokenize()

1. Run Presidio analyze (same path as `redact()`)
2. Apply include/exclude filtering (same logic as `redact()`)
3. Sort entities by start position
4. Assign per-type counters (left-to-right)
5. Replace each match with `<TYPE_N>`, record mapping
6. Return `TokenizeResult`

### deanonymize()

1. Accept text + mapping dict
2. Sort tokens longest-first (prevents `<EMAIL_ADDRESS_1>` matching inside `<EMAIL_ADDRESS_10>`)
3. `str.replace(token, original_value)` for each entry
4. Return restored string

### Strict matching

`deanonymize()` uses exact string matching only. If the LLM mangles a token (wrong case, missing brackets), it stays as-is. This is deliberate — predictable behavior over magic.

## Edge Cases

| Case | Behavior |
|------|----------|
| Multiple entities of same type | Numbered left-to-right: `<EMAIL_ADDRESS_1>`, `<EMAIL_ADDRESS_2>` |
| Same value appears twice | Gets different tokens (each occurrence is independent) |
| Overlapping entities | Presidio resolves overlaps before tokenization |
| Empty text | Returns `TokenizeResult(text="", mapping={}, entities=[])` |
| No entities found | Returns original text with empty mapping |

## Not In Scope

- Session consistency (same value → same token across calls)
- Fake data replacement (synthetic names instead of tokens)
- Streaming / chunk-aware tokenization
- Mapping encryption or access control
